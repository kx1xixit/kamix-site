How we brought kxAI to Kamix

February 21, 2026 Kane
blog

We brought kxAI, a local LLM service, to Kamix (1.0.0 early access) as a proof of concept of Kamix services and its modularity. Now, we'll show you how.

1. We set our goals

Firstly, we wanted to prove to the Kamix supporters that our modularity and service architecture actually worked. So, we started setting our goals.

Fun fact: We previously planned to add a Rok integration (Rok is another self-hosted LLM service) prior to creation of our own service, but because their software is a work-in-progress, it wasn't used and kxAI was created.

2. We set up the server

Initially, we went with a Python + Ollama setup, but then used LM Studio as our backend as it was simpler and natively supported API keys. Then, we set up authentication and prompted our first LLM, which is Qwen 2.5¬†(which is also the default and only model you'd be using if you ran kxAI on Kamix).

We shared a few API keys with our contributors and let them get to work, testing our service to ensure it worked before even starting work on the kxAI Integration service for Kamix.

3. We created a draft

Before writing the actual service, we checked if it was even possible. While it did add another extension (PenguinMod's HTTP extension) to an already large list, it was worth it to bring kxAI to Kamix.

We wrote a simple script that prompts kxAI, and when it worked, we went to work on coding the actual service.

4. We went to work

We imported Kamix, copied "bases"¬†(which is the "base" service, serving as a template for all Kamix services), and made three core commands:

- kxai setup : set your kxAI API key

- kxai ask : ask kxAI something

- kxai help : Pretty much the list you just saw!

How you can get your own API key

Because we don't plan to create an automated script to create an API key, you'll need to message @kx1xixit on Discord, and wait for them to generate you an API key.¬†Please note that MCP is disabled and @kx1xixit can not show you your API key after it is generated.
